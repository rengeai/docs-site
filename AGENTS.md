# AGENTS.md

本文件用于定义进入本仓库工作的所有 AI 代理人应遵循的基础原则。

## 适用范围

- 本文件适用于仓库根目录及其所有子目录
- 所有进入本仓库执行阅读、分析、编写、修改、提交、审查任务的 AI 代理，都应先阅读本文件
- 若子目录中存在更具体的 `AGENTS.md`，则子目录文件优先于本文件

## 进入仓库后的第一动作

- 先阅读 `AGENTS.md`
- 先理解项目目标、文档结构和既有约定，再开始修改
- 非必要不要直接改动文件
- 不明确时优先补充上下文，而不是立即做假设

## 基本工作原则

- 全程使用中文简体沟通与编写文档，除非必须保留英文术语
- 优先沿用现有术语、结构和命名，不随意创造新词
- 优先做小步、清晰、可审计的修改
- 修改前先确认影响范围，避免无关改动
- 不要混合不同意图的改动
- 未经用户明确，不要在输入任何隐私或者个人信息

## 文档修改原则

- 优先保持文档结构稳定
- 新增内容时，优先补入已有章节体系；只有在确有必要时再新增文档
- 单篇文档尽量控制在 1000 字以内；如果超过 1000 字，应优先考虑拆分为多篇文档
- 尽量使用中文表达，除非是非常常见且必须保留的术语，例如 `GitLab`、`Issue`、`MR`、`CEO`、`OKR`、`PMP`
- 对外显式命名优先使用中文，例如角色名、流程名、状态名
- 示例文档要明确场景边界，不混合多个不同流程

## Git 与提交原则

- 每次完成一个清晰意图的修改后，做一次最小提交，并推送到远程仓库
- 相同意图的修改合并为一次提交
- 不同意图的修改拆分为多次提交
- Git 提交信息使用英文
- 未经明确要求，不要改写历史，不要执行破坏性 Git 操作

## 修改代码或文档前的检查

- 先查看相关文件现状
- 先确认是否已有相同或相近内容
- 先确认当前工作区是否干净，避免误提交无关修改
- 如果发现用户已有未提交修改，默认不要覆盖或混入

## 输出与协作原则

- 回答要直接、清晰、可执行
- 对关键规则、结构和术语保持前后一致
- 如果某项能力依赖具体 AI 工具支持，应明确说明
- 如果无法自动生效，也要给出可落地替代方案

## 关于自动读取的说明

- 支持 `AGENTS.md` 约定的 AI 工具，应在进入仓库后优先读取本文件
- 如果某个 AI 工具本身不支持自动读取 `AGENTS.md`，则应在启动任务时显式将本文件内容作为工作指令输入

## 本仓库当前优先约定

- 项目管理方法论采用 `PMP`
- 目标管理采用 `OKR`
- 日常协作与交付工具采用 `GitLab`
- 组织、权限、Issue、里程碑、MR 等机制优先沿用 GitLab 语义
